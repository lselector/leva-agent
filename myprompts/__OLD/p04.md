## 1. Frontend вҖ” Vanilla HTML/CSS/JS SPA

### Tech: Plain HTML + CSS + JavaScript

**Why vanilla:**
- Zero build step, zero dependencies
- No Node.js or npm required
- Served as static files by FastAPI
- Easy to understand and modify
- Works in any browser

**Single-page layout (tab-based navigation):**

| Tab | Purpose |
|-----|---------|
| Chat | Main conversation, streaming tokens, markdown rendering |
| Files | Upload / browse workspace files |
| Jobs | View running & scheduled jobs |
| Settings | Model selection, preferences |

**Key features:**
- Streaming chat via SSE using `fetch()` +
  `ReadableStream` (native browser APIs)
- Markdown rendering via a small vanilla JS
  markdown parser (no library needed, or use
  a single `marked.min.js` from CDN)
- File upload via `<input type="file">` +
  `FormData` + `fetch()`
- Chat history sidebar (list of sessions)
- Responsive layout, works on laptop
- All styles in one `styles.css` file
- No inline styles anywhere

**Directory:**
```
frontend/

в”ңв”Җв”Җ styles.css            вҶҗ ALL styles here

в”ңв”Җв”Җ chat.js               вҶҗ chat + SSE streaming
```

**How it's served:**
FastAPI (Server A) mounts the `frontend/`
directory as static files:

```python
from fastapi.staticfiles import StaticFiles

    "/",
    StaticFiles(
        directory="frontend",
        html=True

    name="frontend",

```

User opens `http://localhost:8000` in browser
and gets the SPA. No separate dev server needed.

## 2. Server A вҖ” LLM Gateway (port 8000)

### Tech: FastAPI + uvicorn

**Responsibilities:**
- Serve the frontend static files
- Accept user messages from the frontend
- Manage conversation history (short-term memory)
- Stream LLM responses back via SSE
- Handle OpenAI tool-call loop (multi-step)
- Dispatch tool calls to Server B when needed
- Log all interactions to Layer 2 memory
- Serve chat history / session management

**Endpoints:**

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/chat` | Send message, get SSE stream back |
| GET | `/api/chat/history` | List past sessions |
| GET | `/api/chat/{session_id}` | Get messages for a session |
| DELETE | `/api/chat/{session_id}` | Delete a session |
| GET | `/api/models` | List available models |
| PUT | `/api/models/current` | Switch active model |
| GET | `/api/health` | Health check |
| GET | `/*` | Serve frontend static files |

**Key design decisions:**
- All API routes prefixed with `/api/` to avoid
  conflicts with static file serving
- SSE streaming (not WebSocket) for simplicity;
  the client sends POST, server responds with
  `text/event-stream`
- Tool-call loop runs server-side: when the LLM
  returns a tool_call, the gateway executes it
  (either locally for memory tools, or by calling
  Server B for automation) and feeds the result
  back to the LLM, continuing until the LLM
  produces a final text response
- Session state stored in-memory + persisted to
  `memory/sessions/` as JSON files
- Soul files (Layer 1) loaded on every turn
  (same as current behavior)

**Directory (inside src/):**
```

в”Ӯ   в”ңв”Җв”Җ __init__.py
в”Ӯ   в”ңв”Җв”Җ app.py            вҶҗ FastAPI app + static mount
в”Ӯ   в”ңв”Җв”Җ routes_chat.py    вҶҗ /api/chat endpoints
в”Ӯ   в”ңв”Җв”Җ routes_models.py  вҶҗ /api/models endpoints
в”Ӯ   в”ңв”Җв”Җ streaming.py      вҶҗ SSE helper
в”Ӯ   в”ңв”Җв”Җ session_store.py  вҶҗ session persistence
в”Ӯ   в””в”Җв”Җ tool_dispatch.py  вҶҗ route tool calls
```

**What moves from current code:**
- `agent_loop.py` logic вҶ’ `routes_chat.py`
  + `tool_dispatch.py`
- `models.py` вҶ’ stays, used by server_llm
- `memory_store/short_term.py` вҶ’ enhanced into
  `session_store.py` (per-session buffers)
- `config.py` вҶ’ stays, add server config
- `tools/registry.py` вҶ’ stays, used by both

## 6. Startup Script (`start`)

Updated `start` script launches both servers:

```bash
#!/usr/bin/env bash
# Start Jarvis: LLM Gateway + Automation Engine

# Start LLM Gateway (port 8000, serves frontend)
uvicorn src.server_llm.app:app \
  --host 127.0.0.1 --port 8000 &
PID_LLM=$!

# Start Automation Engine (port 8001)
uvicorn src.server_auto.app:app \
  --host 127.0.0.1 --port 8001 &
PID_AUTO=$!

echo "Jarvis is running:"
echo "  Open in browser: http://localhost:8000"
echo "  Automation API:  http://localhost:8001"

# Trap Ctrl+C to kill both
trap "kill $PID_LLM $PID_AUTO" EXIT
wait
```

User just runs `./start` and opens browser to
`http://localhost:8000`. That's it.

### SSE Streaming in Vanilla JS

```javascript
// chat.js вҖ” streaming via fetch + ReadableStream
async function sendMessage(text, sessionId) {
  const resp = await fetch("/api/chat", {
    method: "POST",
    headers: {"Content-Type": "application/json"},
    body: JSON.stringify({
      message: text,
      session_id: sessionId
    })
  });
  const reader = resp.body.getReader();
  const decoder = new TextDecoder();
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    const chunk = decoder.decode(value);
    // parse SSE lines, append to chat
    appendToChat(chunk);
  }
}
```

No libraries needed. Works in all modern browsers.

## 9. Data Flow Example: User Sends a Message

```
1. User types in Chat view вҶ’ POST /api/chat
   { "message": "Read my Gmail inbox",
     "session_id": "abc123" }

2. LLM Gateway (Server A):
   a. Load soul files (Layer 1)
   b. Load session history from session_store
   c. Build messages array
   d. Call OpenAI chat completions (streaming)
   e. LLM returns tool_call: browser_gmail_inbox
   f. Gateway POSTs to Server B:
      POST http://localhost:8001/browser/gmail/inbox
   g. Server B launches Playwright, logs into
      Gmail, extracts inbox, returns JSON
   h. Gateway feeds tool result back to LLM
   i. LLM produces final text response
   j. Gateway streams tokens to frontend via SSE
   k. Gateway saves session to memory/sessions/

3. Frontend renders streamed markdown response
```

## Summary

This plan converts Jarvis from a CLI agent into
a modern local-first architecture with:

1. **Vanilla HTML/CSS/JS SPA** вҖ” clean chat UI
   with streaming, file upload, job monitoring.
   No Node, no npm, no build step. Served as
   static files by FastAPI.
2. **LLM Gateway (Server A)** вҖ” handles all AI
   logic, streaming, tool orchestration, memory.
   Also serves the frontend.
3. **Automation Engine (Server B)** вҖ” file ops,
   Playwright browser automation, scheduled jobs.

All Python. All local. User runs `./start` and
opens `http://localhost:8000` in their browser.
The CLI channel remains as a fallback.
Implementation is split into 6 phases, starting
with a minimal skeleton and progressively
adding features.
