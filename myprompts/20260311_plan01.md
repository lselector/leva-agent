# Plan: Email Operations & Web Research
## 2026-03-11

Based on the Perplexity thread discussion and
analysis of our current Jarvis codebase.

---

## Context & Problem

We tried Playwright browser automation for
Gmail — it was **slow and unreliable**:
- Gmail takes 30-60s to load in Playwright
- CSS selectors break when Gmail updates
- Google detects automation and blocks it
- Requires exclusive Chrome profile access
- Cannot run headless reliably

The Perplexity thread discusses using **Chrome
DevTools Protocol (CDP)** directly from Python
as a faster alternative, and also mentions
how OpenClaw handles web operations.

---

### What we have:
- Two-server architecture (LLM :8000, Auto :8001)
- Playwright-based browser_manager.py
  (uses `launch_persistent_context` with

## Recommended Approach: Dual Strategy

### Strategy A: Gmail API (for email)
### Strategy B: CDP from Python (for web)

**Why two strategies:**
- Email is best handled via official Gmail API
  (fast, reliable, Google-approved)
- Web research/browsing is best handled via
  CDP connecting to your running Chrome
  (fast, uses your logged-in sessions)

## Part 1: Gmail via Official API

### Why Gmail API > Playwright for email:
| Feature | Playwright | Gmail API |
|---------|-----------|-----------|
| Speed | 30-60s | < 1 second |
| Reliability | Fragile | Rock solid |
| Detection | Blocked | Approved |
| Headless | Broken | Works fine |
| Auth | Chrome profile | OAuth2 |

### Implementation Steps:

#### Step 1.1 — Install Gmail API packages
```bash
uv add google-api-python-client \
       google-auth-httplib2 \
       google-auth-oauthlib
```

#### Step 1.2 — Google Cloud setup
1. Create project at console.cloud.google.com
2. Enable Gmail API
3. Configure OAuth consent screen
4. Create OAuth2 Desktop credentials
5. Download `credentials/gmail_credentials.json`

#### Step 1.3 — Create `src/gmail_api/` package
```
src/gmail_api/
  __init__.py
  auth.py       ← OAuth2 flow + token mgmt
  client.py     ← Gmail API operations
  routes.py     ← FastAPI endpoints
```

**auth.py** — handles:
- Load credentials from JSON
- Check for existing token
- Refresh expired token
- Browser auth flow if no token
- Save token for reuse

**client.py** — operations:
- `get_inbox(max_results=15)`
- `get_email(msg_id)` — full email
- `send_email(to, subject, body)`
- `search_emails(query)`
- `get_labels()`

**routes.py** — endpoints:
- `POST /api/gmail/inbox`
- `POST /api/gmail/send`
- `POST /api/gmail/search`
- `GET  /api/gmail/email/{id}`

#### Step 1.4 — Wire into Server B
Add routes to `src/server_auto/app.py`:
```python
from src.gmail_api.routes import (
    router as gmail_router
)
app.include_router(gmail_router)
```

#### Step 1.5 — Register as LLM tools
Add to `src/tools/registry.py`:
- `gmail_inbox` — read recent emails
- `gmail_send` — send an email
- `gmail_search` — search emails

This lets the LLM call Gmail tools directly
from chat: "Check my inbox" or "Send email
to John about the meeting".

#### Step 1.6 — First-time auth
```bash
python -m src.gmail_api.auth
```
Opens browser → Google login → grant access →
saves `credentials/gmail_token.json`.
One-time setup, token auto-refreshes.

#### Step 1.7 — Test
```bash
curl -s -X POST \
  http://localhost:8001/api/gmail/inbox \
  | python -m json.tool
```

## Part 2: Web Research via CDP

### Why CDP > Playwright for web browsing:
- Connects to your **already-running** Chrome
- Uses your existing login sessions
- No need to close Chrome first
- Much faster (no browser launch overhead)
- Less likely to be detected as automation
- Python library: `pychrome` or raw websocket

### How it works:
1. Start Chrome with remote debugging:
   ```bash
   /Applications/Google\ Chrome.app/\
   Contents/MacOS/Google\ Chrome \
     --remote-debugging-port=9222
   ```
   Or add to your Chrome shortcut/alias.

2. From Python, connect via CDP:
   ```python
   import pychrome
   browser = pychrome.Browser(
       url="http://127.0.0.1:9222"
   )
   tab = browser.list_tab()[0]
   tab.start()
   tab.Page.navigate(url="https://...")
   ```

### Implementation Steps:

#### Step 2.1 — Install CDP Python package
```bash
uv add pychrome
```

#### Step 2.2 — Create `src/cdp_browser/` package
```
src/cdp_browser/
  __init__.py
  client.py     ← CDP connection manager
  actions.py    ← High-level browser actions
  routes.py     ← FastAPI endpoints
```

**client.py** — CDP connection:
- Connect to Chrome on port 9222
- List open tabs
- Create new tabs
- Execute JavaScript
- Navigate pages
- Take screenshots
- Extract page content

**actions.py** — high-level operations:
- `web_search(query)` — Google search,
  extract results
- `web_fetch(url)` — fetch page, convert
  to markdown/text
- `web_extract(url, selector)` — extract
  specific content
- `web_screenshot(url)` — take screenshot

**routes.py** — endpoints:
- `POST /api/web/search` — web search
- `POST /api/web/fetch` — fetch URL content
- `POST /api/web/screenshot` — screenshot
- `GET  /api/web/tabs` — list open tabs

#### Step 2.3 — Web Search tool
The `web_search` action should:
1. Open Google in a new tab
2. Type the search query
3. Wait for results
4. Extract top 5-10 results (title, URL,
   snippet) via JavaScript
5. Close the tab
6. Return structured JSON

#### Step 2.4 — Web Fetch tool
The `web_fetch` action should:
1. Navigate to the URL
2. Wait for page load
3. Extract main content via JavaScript
   (strip nav, ads, footer)
4. Convert to clean text/markdown
5. Truncate to reasonable size (10K chars)
6. Return the content

#### Step 2.5 — Register as LLM tools
Add to `src/tools/registry.py`:
- `web_search(query)` — search the web
- `web_fetch(url)` — fetch page content
- `web_screenshot(url)` — take screenshot

This lets the LLM do research:
"Search for Python CDP libraries" or
"What does this webpage say?"

#### Step 2.6 — Chrome launch helper
Create a script `chrome_debug` to start
Chrome with debugging enabled:
```bash
#!/usr/bin/env bash
/Applications/Google\ Chrome.app/\
Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 &
```

Or better: detect if Chrome is already
running with debugging, and if not, suggest
the user restart Chrome with the flag.

#### Step 2.7 — Test
```bash
# List tabs
curl http://localhost:8001/api/web/tabs

# Search
curl -X POST \
  http://localhost:8001/api/web/search \
  -H "Content-Type: application/json" \
  -d '{"query":"Python CDP library"}'

# Fetch
curl -X POST \
  http://localhost:8001/api/web/fetch \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com"}'
```

## Part 3: LinkedIn via CDP

### Why CDP for LinkedIn:
- LinkedIn has **no public API** for feed
  reading or liking posts
- Browser automation is the only way
- CDP is better than Playwright because:
  - Uses your already-running Chrome
  - Uses your existing LinkedIn login
  - No need to close Chrome first
  - Faster and less detectable

### Implementation Steps:

#### Step 3.1 — LinkedIn CDP actions
Add to `src/cdp_browser/actions.py`:
- `linkedin_feed()` — get recent feed posts
- `linkedin_like(post_index)` — like a post
- `linkedin_connections()` — list connections
- `linkedin_profile(name)` — view a profile

#### Step 3.2 — LinkedIn routes
Add to `src/cdp_browser/routes.py`:
- `POST /api/linkedin/feed` — get feed
- `POST /api/linkedin/like` — like a post
- `GET  /api/linkedin/connections` — list
- `POST /api/linkedin/profile` — view profile

#### Step 3.3 — Register as LLM tools
Add to `src/tools/registry.py`:
- `linkedin_feed()` — see recent posts
- `linkedin_like(post_index)` — like a post

This lets the LLM handle LinkedIn from chat:
"Show me my LinkedIn feed" or
"Like the latest post from John".

#### Step 3.4 — Migrate from Playwright
Replace the current Playwright-based
`linkedin_actions.py` with CDP-based code.
The CDP version:
1. Connects to your running Chrome
2. Opens LinkedIn feed tab (or reuses one)
3. Extracts posts via JavaScript
4. Clicks like buttons via CDP
5. No browser launch overhead

#### Step 3.5 — Test
```bash
# Get feed
curl -X POST \
  http://localhost:8001/api/linkedin/feed \
  | python -m json.tool

# Like a post (dry run)
curl -X POST \
  http://localhost:8001/api/linkedin/like \
  -H "Content-Type: application/json" \
  -d '{"post_index":0,"dry_run":true}'
```

## Part 4: Alternative — Perplexity API
### (for web search without browser)

If CDP web search proves unreliable, we can
use the **Perplexity API** as a fallback:

```bash
uv add requests  # or use httpx
```

```python
import httpx

resp = httpx.post(
    "https://api.perplexity.ai/chat/completions",
    headers={
        "Authorization": f"Bearer {PPLX_KEY}",
    },
    json={
        "model": "sonar",
        "messages": [
            {"role": "user",
             "content": query}
        ],
    },
)
```

This gives web-grounded answers without
any browser automation. Fast and reliable.

Register as tool: `web_research(query)`

## Summary

This plan replaces the slow/broken Playwright
Gmail automation with two better approaches:

1. **Gmail API** — official, fast, reliable
   OAuth2-based email operations
2. **CDP from Python** — connect to your
   running Chrome for web research, using
   your existing login sessions

Both are registered as LLM tools so Jarvis
can read email and search the web directly
from chat conversations. The Perplexity API
serves as a fallback for web search if CDP
proves unreliable.

This gives Jarvis two of the most requested
capabilities from the popular_skills list:
- **#1 Web Search**
- **#4 Email Assistant**
